# 재설계 A — 공개벤치 중심 벤치마크 (작업 목록)

> **배경**: 내부 코퍼스 결과는 §10.9에서 **88%가 자동생성 near-duplicate**(TTR 0.156)로 밝혀져, 기법 차이와 코퍼스 아티팩트가 섞였다. 유형별 n=5~12로 통계력도 없다. → **공개벤치로 축을 옮겨 재측정**한다.
> **유지**: 아키텍처(`RagBackend`·하니스)·기법 어댑터·방법론(통제·자기반증·강judge)은 **그대로 재사용**. 다시 짓는 건 *시스템이 아니라 실험*이다.

## ★ A의 핵심 이득
- **통계력** — 데이터셋당 수천 문항 확보 가능 → 유의성 검정이 실제로 성립.
- **EM/F1 병행 측정** → **논문 수치와 직접 비교 가능**(§10.11 "메트릭 달라 비교 불가" 갭 해소).
- **재현성·비교가능성** — 남이 검증 가능. "자체 코퍼스 과적합" 의심 제거.
- **한계(명시)** — 영어·위키 도메인이라 원 목표(한국어 조직문서 연결성)에서 벗어남 → **결론 적용범위를 명시**하고, 조직문서 축은 별도 트랙(B/C)으로 남긴다.

## Phase 1 — 벤치 선정·확보
- [ ] **1.1 데이터셋 확정** — 질문 유형 축을 공개 데이터로 커버
  - 다중홉: **HotpotQA**(확보됨) · **MuSiQue** · **2WikiMultihopQA**
  - 단일사실: **NQ** 또는 **PopQA**(엔티티 중심)
  - 전역·종합(abstract): 공개셋 드묾 → VLDB 논문 방식(LLM 생성 125문항/셋) 차용 여부 결정
- [ ] **1.2 변환기 일반화** — 기존 HotpotQA 변환 로직을 **범용 스크립트**로(HF datasets → `data/<bench>/` 코퍼스 + `config/eval_<bench>.yaml`)
- [ ] **1.3 표본 크기 결정** — 데이터셋당 **n≥200**(유형별 유의성 확보). power 근거 기록

## 선행 결정 필요 (비용 스코핑)
공개벤치 × 기법 × n=200 은 **LLM 비용·시간이 곱연산**으로 커진다. 착수 전 확정:
1. **데이터셋 몇 개?** (2개 권장: 다중홉 1 + 단일사실 1 → 축은 커버, 비용 절반)
2. **그래프 계열 전부? 대표만?** (그래프는 벤치마다 **재인덱싱** 필요 → 가장 비쌈. `e2b_hybrid` 대표 1개 권장)
3. **n=200 vs 500** (검정력 vs 비용)

> 권장 최소 구성: **HotpotQA + PopQA(or NQ) × {standard, bm25, hybrid, e2b_hybrid, hipporag} × n=200** — 유형 2축·기법 5종·통계 성립.
