# GraphRAG 그래프 품질 개선 설계안 (구상)

> 현재 GraphRAG가 저성능인 원인은 **방법론이 아니라 빈약한 그래프 구성**(엔티티 이름만·노이즈 다수·type/description/임베딩 없음·커뮤니티 요약 없음)이다(CLAUDE.md §10). 본 문서는 구현 전 단계로, 그래프 품질을 높이는 기법을 조사·비교하고 우리 프로젝트에 맞는 설계를 제안한다. (작성: 2026-06-30)

## 0. 진단 요약 (현재 그래프의 문제)
- 엔티티 4907개 — 이름만 존재, `"2024-01-02"`·`"Hr-006"` 같은 **노이즈 엔티티 다수**
- 엔티티 **type·description 없음**, **임베딩 전부 None**(벡터 노드 검색 불능)
- 관계는 평면 트리플(`["계약직 직원","심사 통과","정규직 전환"]`) — **관계 설명 없음**
- **커뮤니티 탐지·요약 미구현**(GraphRAG 전역 강점의 핵심)

## 1. 개선 레버 (기법 × 트레이드오프)

### L1. 추출기 업그레이드 (가장 직접적)
| 추출기 | 특징 | 평가 |
| --- | --- | --- |
| `SimpleLLMPathExtractor` (현재) | 평면 트리플, 속성 없음 | ❌ 노이즈·빈약 |
| `DynamicLLMPathExtractor` | 엔티티 **type + 속성(description)** 동적 추출 | 유연, 스키마 자동 발견 |
| `SchemaLLMPathExtractor` | **도메인 스키마 강제**(엔티티/관계 타입·속성, `strict`) | ✅ 노이즈 최소·정밀 |

- LlamaIndex 네이티브로 두 추출기 모두 `allowed/possible_entity_props`·`relation_props`를 지원 → 엔티티 description·관계 description 부여 가능.

### L2. 엔티티 정규화 / 중복제거 (Entity Resolution)
- 같은 대상의 다른 표기를 한 노드로 병합(예: "급여 규정" = "HR-006", "J. Smith" = "John Smith").
- 날짜·코드·일반 구절 등 **노이즈 엔티티를 필터링**(의사결정에 의미 있는 엔티티만 그래프화 — "모든 명사를 노드로 만들지 말 것").

### L3. 엔티티 임베딩 실제 적용
- 현재 엔티티 임베딩이 전부 None인 문제를 수정 → 벡터 기반 노드 검색(`VectorContextRetriever`) 복구.

### L4. 문서 참조 그래프 (★우리 코퍼스의 강점)
- 본문의 **「문서명」 인용을 결정론적으로 파싱**해 **문서↔문서 엣지**를 직접 생성.
- LLM 불필요·노이즈 0·100% 정확. "문서 간 연결"이라는 프로젝트 목표를 가장 싸고 정확하게 표현.
- 우리 코퍼스는 모든 규정/문서가 「연차 유급휴가 규정」 식으로 서로를 명시 인용 → 사실상 정답 연결 그래프가 텍스트에 이미 존재.

### L5. 커뮤니티 탐지 + 요약
- Leiden 커뮤니티 + 커뮤니티별 LLM 요약 → GraphRAG의 **전역 sensemaking** 핵심.
- LlamaIndex `PropertyGraphIndex` 기본엔 없음 → 커스텀 구현 또는 공식 `microsoft/graphrag` 라이브러리 채택 필요.

### 대안 기법
- **LightRAG** — 듀얼레벨(로컬 이웃 / 글로벌 키워드) 검색, 엔티티+관계 description, **검색 비용 급감**(GraphRAG 대비). 커뮤니티 없이 효율적.
- **HippoRAG2** — Personalized PageRank 기반 다중 홉 — 우리 약점(multi)에 직접 대응.

## 2. 추천 설계 (우리 프로젝트 맞춤, 단계적)

```text
[1] 도메인 스키마 정의 → SchemaLLMPathExtractor
[2] 문서참조 그래프 결합 (「문서명」 파싱 → 결정론적 doc-doc 엣지)   ← 싸고 확실
[3] 엔티티 임베딩 수정 + 정규화/노이즈 엔티티 필터
[4] (그다음) 커뮤니티 요약 → 전역 질문  /  또는 LightRAG 별도 기법 추가
```

핵심: 노이즈투성이 자동 트리플(현재) 대신 **① 도메인 스키마로 깨끗하게 추출 + ② 문서 인용을 직접 파싱해 연결성을 확정**. 이 둘을 합치면 "문서 관계 그래프"가 실제로 만들어진다.

### 2.1 도메인 스키마 초안 (하울 코퍼스)
- **엔티티 타입**: 직원, 부서, 프로젝트, 규정/문서, 양식, 장비, 회의, 법령
- **관계 타입**: 소속(직원-부서), 참여(직원-프로젝트), 사용(프로젝트-장비), 담당(직원-장비), 보고(부서-보고서), 근거/인용(문서-문서), 승인/전결(금액-권한)
- **엔티티 속성**: name, type, description(역할·정의)
- **관계 속성**: description

### 2.2 중요도 기반 티어 추출 (문서 특성별 차등)
> 모든 문서를 똑같이 비싸게 추출하지 않고, **문서 중요도/특성에 따라 추출 강도를 차등**한다. 데이터 거버넌스의 "마스터 데이터 vs 트랜잭션 데이터" 차등 관리와 같은 원리.

| 문서군 | 중요도 | 추출 방식 | 이유 |
| --- | --- | --- | --- |
| 규정·법률(HR/AD/DP, 근로기준법 등) | 높음 | **Schema(강제·검증)** + 참조 파싱 | 권위 있는 정답 원천 → 정밀·노이즈0 |
| 통계·사업계획·직원현황 | 중간 | **Dynamic**(선택) | 구조 있으나 유연성 허용 |
| 월간보고·진행보고·회의록·공지 | 낮음 | **참조 파싱(L4) 위주**, (선택) 평면 | 휘발성·다수 → 비용↓, 깊은 추출 불필요 |

**핵심 통찰**: 저중요 문서(보고서·공지)의 그래프 가치는 *내부 엔티티*가 아니라 **다른 문서를 인용하는 엣지**(「규정명」)다. 따라서 깊은 엔티티 추출 대신 **「문서명」 참조만 결정론적으로 파싱**하면 충분하고 LLM도 거의 안 쓴다.

**⚠️ 필수 제약 — 티어 간 엔티티 정규화**: 티어가 달라도 같은 대상은 **하나의 노드로 병합**돼야 한다(보고서의 `연차 유급휴가 규정` = 규정 문서의 그 노드). 안 그러면 끊긴 서브그래프가 되어 "보고서↔규정" 연결이 깨진다 → **공통 엔티티 namespace/정규화** 필수.

**권장 2-티어**:
1. 규정·법률 → Schema 추출(엔티티+속성+관계) + 참조 파싱
2. 나머지 → 참조 파싱(L4) 위주 (+ 필요시 Dynamic 가볍게)

→ *권위 문서는 정밀한 관계 그래프, 휘발성 문서는 인용 링크로 연결* — 비용·품질 동시 최적. (실효성은 구현 후 같은 평가 하니스로 검증)

## 3. 검증표 (어떤 레버가 무엇을 고치나)
| 레버 | 고치는 문제 | 기대 효과 |
| --- | --- | --- |
| L1 스키마 추출 | 이름만/노이즈 엔티티 | 깨끗한 type+설명 노드 |
| L2 정규화 | 중복·노이즈 엔티티 | 노드 병합·연결 강화 |
| L3 임베딩 수정 | 벡터 노드검색 불능 | recall 회복 |
| L4 문서참조 그래프 | 문서 연결 부재 | multi·관계형 직접 개선 |
| L5 커뮤니티 요약 | 전역(global) 약함 | global 답변 품질 개선 |
