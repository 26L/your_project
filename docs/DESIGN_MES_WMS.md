# MES · WMS — 개념과 RAG 연결 관점

> 제조·물류 현장 시스템(MES·WMS)이 무엇이고, 왜 이 프로젝트의 **문서 연결성 RAG**(특히 커뮤니티 프로필)와 잘 맞는지 정리. 학습·기획용. RAG 결론은 [RESULT_SUMMARY.md](RESULT_SUMMARY.md)·[RESULT_PROFILES.md](RESULT_PROFILES.md) 참고.

## 1. MES (Manufacturing Execution System, 생산관리시스템)

**정의**: ERP(계획)와 현장 설비(제어) **사이**에서 생산을 **실시간 실행·감시**하는 시스템. "계획을 실제 생산으로" 옮기는 층.

**표준**: **ISA-95**(기업-제어 통합 계층), ISA-88(배치). 기능 분류 **MESA-11**.

**핵심 기능·데이터**
| 기능 | 생성 문서·데이터 |
| --- | --- |
| 작업 지시(dispatching) | 작업지시서, 라우팅, BOM |
| 생산 추적(tracking) | **로트/배치 genealogy(계보)**, 실적 |
| 품질 관리 | 검사기록, 부적합(NCR), CAPA |
| 설비 관리 | 가동데이터·OEE, 정비이력 |
| 문서 관리 | SOP·작업표준(WI) |
| 인력 관리 | 작업자 배정·자격 |

## 2. WMS (Warehouse Management System, 창고관리시스템)

**정의**: 창고의 **입고→적치→보관→피킹→포장→출고→재고**를 관리하는 시스템.

**핵심 기능·데이터**
| 기능 | 생성 문서·데이터 |
| --- | --- |
| 입고(receiving) | 입고지시, ASN(사전출하통지) |
| 적치·보관 | 로케이션(bin) 맵, 재고현황 |
| 피킹·출고 | 피킹리스트, 출고지시 |
| 재고관리 | 실사(cycle count), **로트/시리얼 추적** |

## 3. 왜 RAG, 특히 **그래프/연결성**과 맞는가 ★

MES·WMS 기록은 **본질적으로 촘촘히 연결된 참조망**이다 — 우리 프로젝트가 노린 "문서 연결성"의 **진짜 실무 사례**.

```
작업지시 ──uses──► BOM ──consumes──► 로트(원자재) ◄──입고── 공급사
   │                                    ▲
 runs on ▼                          (WMS 재고)
   설비 ──operated by──► 작업자        │
   │                                    ▼
 produces ▼                         출고 ──ships──► 고객
   완제품 로트 ──inspected──► 품질기록
```

- **다중 홉이 자연스럽다**: *"이 불량 완제품 로트가 쓴 원자재는 어느 입고분·어느 공급사인가?"* → 완제품→BOM→소비로트→입고→공급사(**4~5홉**). 로트 genealogy = 전형적 multi-hop.
- **관계·인과형**: *"이 설비 정지가 어느 주문 지연에 영향?"* → 설비→작업지시→주문. (§CausalRAG류 인과 질문 영역)
- **전역·종합(global)**: *"이번 달 A라인 품질 이슈 요약"* → 커뮤니티 요약이 강한 영역.

## 3.1 왜 관계를 더 깊이 다뤄야 하나 — 두 성질: 계층 + 다중연관 ★

MES/WMS 관계는 **단순 pairwise가 아니다.** 두 구조가 동시에 겹쳐서, 각각 다른 기법을 요구한다:

| 성질 | 무엇인가 | MES/WMS 예 | 맞는 RAG 기법 | 방향 |
| --- | --- | --- | --- | --- |
| **계층(hierarchy)** | 위로 추상화되는 트리 | BOM(제품→반제품→부품), ISA-95(전사→라인→셀), 로케이션(창고→존→빈) | **RAPTOR(트리 요약)** · GraphRAG **커뮤니티 요약** | ▲ 위로 |
| **다중연관(N:M)** | 한 노드가 수많은 노드와 얽힘 | 로트↔작업지시, 설비↔주문, 작업자↔공정 | **그래프/PPR(HippoRAG2)** — 다중홉 연상 | ◄► 옆으로 |

- **계층이 필요한 질문**: "이 라인 이번 달 전반 품질은?" → BOM/조직 트리를 **위로 요약**해야 답이 나옴(global·종합). 평면·단일 그래프탐색으론 약함 → **계층 요약(RAPTOR/커뮤니티)** 필요.
- **다중연관이 필요한 질문**: "이 불량 로트가 거친 모든 설비·작업자·원자재는?" → N:M 관계망을 **옆으로 확산**해야 함 → **PPR(HippoRAG) 다중홉**이 본업.
- **★ 그래서 단일 기법으론 부족하다** — MES/WMS는 **위(계층)와 옆(다중연관)을 동시에** 요구하므로, **RAPTOR(계층) + HippoRAG/그래프(다중연관)를 함께** 비교·조합해야 한다. 이게 이 프로젝트가 여러 기법을 나란히 재는 실무적 이유이고, **다음 기법으로 RAPTOR를 붙이려는 근거**(§계층 축 보강).

> 요약: **관계가 깊다 = 계층(▲) × 다중연관(◄►) 이 겹친다.** 두 축을 각각 잘하는 기법이 달라서, 도메인이 복잡할수록 **단일 검색이 아니라 축별 조합**이 답이 된다.

## 4. 우리 RAG 결론의 적용

| 우리 발견 | MES·WMS 적용 |
| --- | --- |
| **연결성↑ ≠ 성능↑, 질문이 연결을 요구할 때만 그래프 값어치** | MES/WMS는 **로트 추적·인과 질문이 실제로 다중홉을 요구** → 그래프 RAG의 값어치가 **진짜로 사는** 도메인 |
| **자동생성 near-duplicate 코퍼스는 이득↓**(§10.9) | MES/WMS는 **실제·다양·진짜 연결된** 기록 → 커뮤니티 프로필 값어치↑ |
| **2 프로필** | 핀포인트 조회(재고·상태)=`hybrid`, 로트 추적·인과·종합=**`graphrag_e2b_hybrid`/HippoRAG2(PPR)** |
| **HippoRAG PPR = 다중홉 연상** | genealogy 추적(A로트→B→C)이 PPR 확산과 정확히 맞음 |

## 5. 주의 (실무 적용 전)

- **정형 데이터 vs 문서**: MES/WMS 상당수는 **관계형 DB의 정형 레코드**다. RAG(텍스트 검색)보다 **SQL/그래프DB 직접 질의**가 맞는 질문도 많음 — RAG는 **비정형(SOP·NCR·작업지시 코멘트·정비노트)** 과 **여러 시스템 교차 종합**에 쓰는 게 적합.
- **하이브리드 구조 권장**: 구조화 조회는 DB, 자연어·종합·근거 설명은 RAG(위 그래프 검색). 두 시스템은 실무에서 **ERP·MES·WMS가 연동**되므로, 교차 참조를 그래프로 표현하면 우리 커뮤니티 프로필과 직결.
- **보안·권한**: 생산·재고 데이터는 민감 → 접근제어·감사가 전제(우리 벤치 범위 밖).

## 6. 기술 확장 — ragbench로 MES/WMS 검증하기 ★

우리 아키텍처(`RagBackend` + 플러그형 백엔드 + 평가 하니스)는 **코퍼스·평가셋만 바꾸면** 새 도메인을 검증한다. MES/WMS 확장에 필요한 실제 작업:

### (1) 정형 레코드 → 문서화 + 결정론적 참조 그래프
- MES/WMS는 대부분 **관계형 DB/CSV**. 각 레코드(작업지시·로트·입고·NCR…)를 **1문서**로 변환.
- **핵심 이점**: 레코드 간 **FK·로트링크가 이미 결정론적 관계** → LLM 추출 없이 **정확한 엣지**를 만든다. 우리 [§10.3 「문서명」 인용 파싱](../CLAUDE.md)을 **FK 파싱으로 대체** → 가상 코퍼스의 "깨끗한 인용"보다 오히려 **진짜이면서 깨끗한** 연결성.
- 확장 지점: [`ingest/loader.py`](../src/ragbench/ingest/loader.py)에 **CSV/DB 커넥터** 추가(레코드→Document, FK→metadata/본문 「참조」). 그래프 계열은 이 참조를 그대로 엣지로 사용.
- → **§10.2 "이상적 상한값" 한계를 실데이터로 넘어서는 길**(로트/FK는 실무에서도 결정론적이라, 우리 가상 인용의 낙관 편향이 없음).

### (2) 평가셋 — 유형별로 MES/WMS 질문 설계
`config/eval_*.yaml` 형식 그대로, 도메인 질문을 유형 태그로:
| 유형 | MES/WMS 예시 질문 | 기대 강자 |
| --- | --- | --- |
| single | "로트 L-2024-0812 현재 재고 위치?" | hybrid |
| **multi** | "불량 로트 X가 쓴 원자재의 공급사?" (완제품→BOM→소비로트→입고→공급사) | **HippoRAG2/그래프** |
| relational | "설비 EQ-07 정지가 지연시킨 주문은?" | 그래프 |
| global | "이번 달 A라인 품질 이슈 요약" | 그래프+커뮤니티요약 |

### (3) 코드 변경 최소 (플러그형 설계 검증)
- **하니스·인터페이스 무변경** — 새 `data/mes_wms/` 코퍼스 + `config/eval_mes.yaml`만 추가하면 전 기법(hybrid·graphrag_e2b_hybrid·HippoRAG2)이 그대로 돈다.
- 유일한 코드 작업 = **로더 커넥터**(정형→문서+참조). 이게 "새 도메인 = 어댑터 한 층"이라는 우리 설계의 실전 검증이 된다.

### (4) 연결성 전용 지표 실측 가능
- 로트 genealogy는 **정답 경로가 명확**(FK 체인) → [§6.1 "관계 경로 정답성·다중홉 도달률"](../CLAUDE.md) 같은 연결성 지표를 **실측**할 수 있다. 가상 코퍼스에선 애매했던 걸 실데이터로 채점 가능.
- 가설: **MES/WMS 다중홉·인과 질문에서 그래프 계열(HippoRAG2 PPR)이 평면을 실제로 앞선다** — 우리 §10.8("질문이 연결을 요구할 때만 그래프 값어치")을 **긍정 사례로 검증**하는 첫 도메인.

### (5) DB 질의와 역할 분담 (하이브리드 아키텍처)
- 정형 조회(재고 수량·상태)는 **SQL/그래프DB 직접 질의**가 정확·빠름 → RAG는 **비정형(SOP·NCR 코멘트·정비노트) + 여러 시스템 교차 종합 + 근거 설명**에 배치.
- 실무 구성: **라우터**가 질문을 정형(→DB) vs 비정형/종합(→RAG) 분기 — 우리 `graphrag_e2b_adaptive`(질의 라우팅)의 확장선.

## 7. 지시 실행 — 사람(자연어) vs 로봇(로봇의 언어) ★ (RAG → VLA)

MES/WMS의 작업지시는 결국 **행위자에게 지시**로 나간다. 같은 상황·같은 RAG 검색 결과라도 **대상에 따라 출력 형식이 다르다** — 여기서 RAG와 VLA(비전-언어-행동)가 만난다.

```
작업지시/상황 ──► RAG(연결성 검색: 절차·제약·근거를 다중홉/계층으로 연결)
                    │
                    ├──► 사람 작업자  : 자연어 지시 + 근거("왜 이 순서·이 문서")
                    └──► 로봇         : 로봇의 언어 = 실행 가능한 액션
                                        (VLA action token / 스킬 API pick·place·move
                                         / PLC·OPC-UA 명령)
```

- **사람**: RAG 생성 답변 그대로 — 절차 요약 + **근거 문서·관계 경로** 노출(설명가능성).
- **로봇**: RAG가 잡은 절차를 **실행형으로 변환** — VLA면 action token(관절·그리퍼), 전통 로봇이면 스킬 호출(move-to/pick/place), 설비면 PLC·OPC-UA 명령.
- **★ VLA+RAG 구조**: RAG = **비파라미터 지식**(무엇을·왜·제약), VLA = **행동 생성**(어떻게 움직여). **같은 검색 결과, 대상별 다른 디코딩** — 입력측 질의 라우터(`adaptive`)의 **출력측 대칭**.

**왜 여기서 그래프/계층 RAG가 결정적인가**
- 로봇 지시엔 **정확한 절차·순서·의존·안전제약**이 필요 → 여러 문서/시스템에 흩어진 근거를 **다중홉(옆) + 계층(위)** 으로 연결해야 안전한 액션이 나온다(§3.1 두 축).
- **환각 = 물리적 사고.** 사람은 이상한 지시를 거르지만 로봇은 그대로 실행 → **근거 정확성·연결 경로 검증**이 사람보다 훨씬 더 중요. 그래서 "출처 청크 + 관계 경로를 함께 반환"(우리 §4.1 연결 근거 노출)이 로봇 지시엔 **필수 안전장치**.
- 관련 연구 방향: 데모/경험 검색(MemER), 매뉴얼·어포던스 검색(SayCan·VoxPoser), **씬그래프 검색**(물체·공간 관계 → 우리 그래프 검색과 직결). *(개념·링크는 별도 VLA+RAG 정리 참고)*

> ⚠️ **위치**: 이 절은 **아키텍처 구상**(미구현). ragbench는 검색·생성 검증 하니스이고, 로봇 액션 디코딩(VLA)·설비 명령(PLC)은 별도 실행층이다. 다만 **RAG가 "정확한 근거로 지시를 만든다"** 는 부분은 이 프로젝트의 검증 대상과 정확히 이어진다.

## 한 줄
**MES·WMS는 "진짜로 연결된 조직 데이터"의 대표 사례** — 로트 추적·인과·종합 질문이 다중홉을 실제로 요구하므로, 우리 **커뮤니티 프로필(그래프+벡터 / PPR)** 의 값어치가 가상 코퍼스보다 오히려 **더 잘 사는** 도메인이다. 단 정형 레코드는 DB 질의와 역할을 나눠야 한다.
